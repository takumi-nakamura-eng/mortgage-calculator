# calcnavi AGENTS

## 開発ルール

このファイルは、このリポジトリで Codex が作業する際の標準ルールです。
ルールは短く、安定していて、実務的に保ってください。

### Core Workflow

- コミット、ブランチ作成、PR 作成のタイミングはユーザーが判断する。
  - Codex は明示的に依頼された場合のみ、コミット、ブランチ作成、PR 用の変更整理や説明文作成を行う。
- 作業目的を分けたいときは、新しいスレッドで扱う。
  - 現在のスレッドでは、その依頼の目的に直接関係する修正のみを優先し、無関係な変更は原則含めない。
- 新しいスレッドで実装や修正を始めるときは、その目的に合ったブランチを必ず作成する。
  - ブランチ名は `codex/<purpose>` を基本とし、番号付きでもよい。例: `codex/01-bolt-length`
- 別目的の作業が見つかった場合は、現在の変更に無理に混ぜない。
  - 必要なら「別スレッドで扱うべき内容」として短く切り分けて提示する。

### Branch Workflow

- 新しいスレッドまたは新しい目的の作業を始めるときは、最新の `main` を取得してから作業ブランチを作成する。
- `main` を最新化しないまま新規ブランチを切らない。
- 作業ブランチが長生きしそうな場合や、`main` 側の更新が進んでいる場合は、差分が小さいうちに早めに `main` を取り込む。
- `app/tools/**/page.tsx`、`lib/data/tools.ts`、`lib/diagrams/registry.tsx`、共有コンポーネントのような競合しやすい共通ファイルを触る前は、`main` 側の差分確認を優先する。
- 既存ページの役割変更や分割を行う場合は、同じファイルに `main` 側で公開日、関連記事、広告、共通 UI の更新が入っていないか先に確認する。

### Pull Request Workflow

- Codex は、明示的に禁止されていない限り、作業完了後に `gh` を用いて PR 作成まで進めてよい。
- `gh` が利用可能で認証済みであれば、PR 作成は標準作業とし、追加確認は不要とする。
- 認証確認では、資格情報やトークンの値を表示せず、利用可否のみ確認する。
- Codex が自動で実施してよいのは、作業ブランチの作成、commit、push、PR 作成までとする。
- `main` / `master` への直接 push は禁止する。
- merge、rebase、squash、branch 削除はユーザーの明示的な指示がある場合のみ行う。
- `gh` による PR 作成に失敗した場合は、その理由を簡潔に報告し、必要に応じて GitHub API または手動作成用 URL を代替手段として提示する。

### SEO / Content Strategy

- 生成AI時代でも Google 検索流入は見込めるが、単純な情報記事は AI に吸われやすい前提で企画する。
- 狙う対象は、AI だけでは完結しにくい実務作業、特に「理解」より「計算・確認・判定」が必要な場面を優先する。
- calcnavi は読ませる記事より使わせるツールを重視し、実務で最後に使う計算・比較・一次判断の受け皿を目指す。
- 優先するのは、実行型ツール、条件分岐が多いページ、根拠つきの一次判断ページ、定義・式・前提・注意点・例題・FAQ が整理されたページである。
- ツールは単体で終わらせず、用途、入力値の意味、計算前提、注意点、関連リンクを必ず添え、補助記事と内部リンクでクラスター化する。
- 「〇〇とは」だけの薄い用語解説、単純な情報まとめ、一般論だけの記事は優先しない。単純な換算ページでも、実務的需要が高く競合が少なく、関連ツール群の入口として機能するなら採用してよい。
- 検索順位や PV、本数だけを追わず、指名されるツール数、表示クエリ数、実際に使われるページ数、再訪されるページ数、実務導線の強さ、広告表示回数、広告クリック率を重視する。
- 一言で言えば、AI に勝つのではなく、AI では終わらない作業を取りにいく。

#### 広告設置

- Google AdSense などのサードパーティ広告は、ユーザーの計算・確認・判定フローを妨げないことを最優先に配置する。
- 広告ユニットはレスポンシブを基本とし、ツールでは結果セクション下、記事では導入後または本文の区切り、関連記事導線の前後など、意図の切れ目に限定して置く。
- 入力フォーム直上、主要な結果表示の直前直後、FAQ の連続読了を妨げる位置、誤タップを誘発しやすいモバイル密集領域には置かない。
- 広告の追加や位置変更は UX と収益の両面で計測し、必要なら A/B テストで検証する。

### E-E-A-T / 信頼性

- 運営者情報では、実名、経歴、資格、実績、専門領域、監修体制を公開し、内容変更時は随時更新する。
- プライバシーポリシー、利用規約、免責事項、編集ポリシー、問い合わせページを整備し、導線を全体フッターや主要ページから辿れるようにする。
- 一次情報、規格、メーカー資料、実務経験のどれに基づくかを利用者が判断できるよう、記事・ツール・運営情報の責任範囲を明確にする。
- 信頼性向上のための追記は、見た目の装飾よりも、更新日、監修情報、根拠、問い合わせ導線の明示を優先する。

### 再訪施策

- リピーター獲得のため、履歴保存のクラウド同期、メールマガジン登録、ニュースレター、SNS 活用など、再訪理由を作る機能を優先的に検討する。
- 再訪施策は、単なる告知よりも、保存した計算条件の再利用、更新通知、関連基準の改定通知など、実務上の継続利用価値を重視する。
- 個人情報や連絡先を扱う機能では、取得目的、保存期間、利用範囲、削除方法を明示し、プライバシーポリシーと整合した運用にする。
- メールや通知導線は任意参加を原則とし、ツール利用自体を不要に阻害しない。

### Measurement / Improvement

- GA4 と Search Console を使い、流入、表示クエリ、主要操作、離脱箇所、広告パフォーマンスを継続的に計測する。
- ツールでは入力開始、計算実行、結果到達、PDF 出力、関連リンク遷移などのイベントを追跡し、記事ではスクロール、FAQ 展開、関連導線の利用を必要に応じて追跡する。
- UI、広告配置、導線、CTA は A/B テストや比較検証を前提にし、収益と UX の両面から定期的に改善する。
- 計測追加時は、個人情報の過剰取得を避け、必要最小限のイベント設計とする。

### Article Rules

- 記事 frontmatter は `title`, `description`, `publishedAt`, `updatedAt`, `category`, `tags`, `sources` を必ず揃える。
- 記事は実務判断に役立つ内容で書く。結論、判断基準、関連する内部リンクを入れる。
- 情報源は一次情報、公的規格、メーカー資料を優先する。URL と確認日を残す。
- 関連記事や関連ツールは、記事の判断や実務フローに直接効くものだけを追加する。
- 新規記事の標準は `heroDiagram: true` を前提にし、Hero でテーマを伝え、本文は説明に集中させる。
- `tags` と `category` はタグページ・カテゴリページ生成、および関連コンテンツへの内部リンク自動化で利用できる粒度に揃える。

### Article Revision Rules

- この節は既存記事の改修時に適用する。`Article Rules` は常設の基本ルールとして扱う。
- 改修前に、主検索意図、想定読者、残すべき既存価値を先に整理する。
- 改善は必要最小限の差分を基本としつつ、検索意図と本文がズレる場合は見出し構成と導線の組み替えを許容する。
- 記事構成は柔軟でよいが、`導入 / 結論または一次判断 / 比較表または整理表 / 計算例 / FAQ / 関連導線` の骨格は原則維持する。
- Hero を使う記事では、本文内に図解セクションを重複配置しない。Hero はテーマ提示、本文は説明と判断材料に役割分担する。
- 記事冒頭には `この記事でわかること` や `結論` などの導入見出しを置く。枠付き UI は必須ではなく、通常見出し + アイコンでもよい。
- 重要な判断、数値、規格・適用範囲の説明は、本文中でどの出典に基づくか自然文で紐づける。末尾の `sources` 一覧だけで完了扱いにしない。
- 記事 SVG や表は 1表示1メッセージを原則とし、説明・注釈・数値を詰め込みすぎない。詳細説明は本文に任せる。
- 記事 SVG は本文、カード、関連表示で共通利用される前提で、主タイトルは読みやすく、主図形と干渉しない位置に置く。
- 記事 Hero は `1枚の導入カード` を基本とし、左にタイトルとメタ情報、右に SVG を置く。背景は白から淡い青へ流れる弱いグラデーションを使い、本文カードも同系統の面でつなげてよい。
- Hero タイトルは左カラム幅を素直に使い、必要以上に狭い文字幅制限を設けない。SVG は右カラム内で中央に収まり、視覚重心も中央寄りに調整する。
- モバイルでは、図解や表をカード表示、横スクロール、アコーディオン表示などに切り替えてよく、可読性と判断速度を優先する。
- 改修後は、検索意図一致、本文と出典の対応、関連導線の自然さ、SVG の簡潔さを自己レビューする。
- 一時的な CSS 回避やページ固有の表示バグ対応は、記事改修ルールとして AGENTS に固定しない。安定した運用ルールだけを書く。

### Tool Rules

- 計算ロジックは `lib/` に純粋関数として置く。式や判定条件を page-level UI に埋め込まない。
- ツール構成は原則として、`app/tools/<link>/page.tsx` をページ枠、`app/tools/<link>/<Name>Calculator.tsx` を UI、`lib/<domain>/<feature>.ts` を計算ロジック、`lib/diagrams/tools/<link>.tsx` を正規 SVG とする。
- ツールの機能や入力負荷に応じて、ステップ分割、モーダル、タブ、折りたたみを使ってよいが、主要な計算導線は見失わせない。
- ツールの `href`、`diagramKey`、図ファイル名は可能な限り link slug に揃える。例: `bolt-length` -> `diagramKey: "bolt-length"` -> `lib/diagrams/tools/bolt-length.tsx`
- ツール図はツールごとに正規ソースを 1 つにする。本体ページ、カード、印刷・出力用ヘルパーは同じ React SVG コンポーネントから派生させる。
- ツールページは `buildMetadata()`、`Breadcrumbs`、`SoftwareApplication` JSON-LD、関連解説導線を標準構成とする。
- ツールページ上部は、パンくずの下に `軽量Hero 1カード` を置き、タイトル・説明・補助ラベル・図解を統合する。図解を Hero の外に独立配置しない。
- ツールの主機能エリアは `入力条件`、`計算結果`、`計算式・PDF出力` の順を基本とし、入力直後に結果へ入れる導線を優先する。
- 入力UIの淡色背景は、外側コンポーネントではなく、実際に入力する `input` / `select` 要素にのみ付ける。
- PDF出力は `A4 1枚に収める方針` を優先し、入力表と結果表の列幅を揃え、末尾のツールURLはクリック可能にする。
- 1つのツールに役割を詰め込みすぎず、別目的の計算は関連ツールへ自然に誘導する。
- `入力条件` ヘッダーや標準免責文など、複数ツールで繰り返す UI 文言は共通コンポーネントに寄せて重複実装を増やさない。
- 結果は数値だけでなく、単位、判定文脈、入力条件を保持する。
- 明示的に必要な場合を除き、履歴、PDF/印刷、関連導線を壊さない。

### Diagram Rules

- 図の正規ソースは `lib/diagrams/tools/` と `lib/diagrams/articles/` に分ける。ツール図と記事図を同じ名前空間で混ぜない。
- `diagramKey` は最終的な図の入口として使い、記事とツールで別々に解決する。記事カードがツール図を直接拾う実装に戻さない。
- 記事の `diagramKey` は可能な限り記事 slug に揃える。例: `/articles/three-threads` -> `diagramKey: "three-threads"` -> `lib/diagrams/articles/three-threads.tsx`
- 本文、カード、関連表示の図は共通の図レジストリ経由で描画する。個別ページに図コンポーネント import を増やし続けない。
- `thumbnailSvg` のような本文からの生 `<svg>` 抽出には依存しない。図の再利用は正規 SVG コンポーネントを基準に行う。

### UI Rules

- 既存のページ構造と `container`、`page-title`、`page-description` などの共有クラスを優先して使う。
- SEO とサイト共通 metadata の判断は `lib/seo.ts` と `lib/site.ts` に集約する。
- パンくず、関連リンク、記事カード、ツールカードは既存の共有コンポーネントを優先して使う。
- UI変更は見た目だけでなく、構造と導線の変更として扱う。
